# Experimental Log: Metagenomic Classification Tools Benchmark for Long Reads

## 1. Tools Evaluated

Thirteen pipelines were tested, spanning four categories:

| Category | Tool | Version | Database Type | Notes |
|---|---|---|---|---|
| Kmer-based | Kraken2 | v2.0.8 | Nucleotide | Includes internal kmer threshold option |
| Kmer-based | Bracken | v2.0.8 | Nucleotide | Bayesian re-estimation on top of Kraken2 |
| Kmer-based | Centrifuge | v1.0.4 | Nucleotide | Uses EM estimation post-processing |
| Kmer-based | CLARK | v1.2.6.1 | Nucleotide | Discriminative kmer approach |
| Kmer-based | CLARK-S | v1.2.6.1 | Nucleotide | Spaced-seed variant of CLARK |
| Mapping-based | MetaMaps | v0.1 | Nucleotide | Uses EM estimation |
| Mapping-based | MEGAN-N | v6.18.5 | Nucleotide | MEGAN-LR with nucleotide DB, LCA post-processing |
| Mapping-based | deSAMBA | v1.1.12 | Nucleotide | Designed for long reads |
| General-purpose mapper | Minimap2 (align) | v2.18 | Nucleotide | Full alignment mode |
| General-purpose mapper | Minimap2 (map) | v2.18 | Nucleotide | Mapping-only mode |
| General-purpose mapper | Ram | v2.1.1 | Nucleotide | Mapping mode only |
| Protein-DB | Kaiju | v1.8.2 | Protein | Protein-level classification |
| Protein-DB | MEGAN-P | v6.18.5 | Protein | MEGAN-LR with LAST aligner + protein DB |

Tools reviewed but excluded: k-SLAM, MetaPhlAn, ConStrains, PathoScope, KrakenUniq, Sigma, CCMetagen, Gotcha (crashed during DB creation or poor long-read performance). MEGAN-LR with DIAMOND aligner was also excluded (worse than MEGAN-P with LAST).

## 7. Read-Level Classification Accuracy (Figure 1 observations)

Figure 1 displays read-level classification accuracy at species and genus levels across all synthetic and mock datasets.

Key observations:
- Species-level accuracy was consistently lower than genus-level accuracy for all tools, as expected
- Kmer-based tools (shown in red) cluster together in performance, with Kraken2 and Centrifuge generally performing best within the category
- Mapping-based tools (shown in blue) exhibited variable performance; deSAMBA was competitive but MetaMaps and MEGAN-N were less consistent
- Protein-database tools (shown in green) consistently had the lowest read-level accuracy
- General-purpose mappers, particularly Minimap2 in alignment mode, achieved similar or better accuracy than the best classification-specific tools on most datasets
- Mean accuracy values (Figure 1c) showed Minimap2 (alignment) at the top, followed closely by kmer-based tools, with protein-DB tools at the bottom

## 8. Read Length vs Classification Accuracy (Figure 2 observations)

Figure 2 compares Q1, median, and Q3 read lengths for true positive versus false positive classifications for each dataset, focusing on Minimap2 (alignment) and Kraken2.

Key observations:
- Longer reads were generally classified more accurately (higher TP rate)
- False positive classifications tended to involve shorter reads
- However, filtering the dataset to retain only the longest reads reduced overall accuracy because read-length distributions differ across species: some species naturally produce shorter reads, and removing them biases the community composition
- PacBio HiFi reads showed tighter length distributions compared to ONT reads

## 15. PCA and Clustering of Real Datasets (Figure 4 observations)

Figure 4 shows 2D PCA projections and hierarchical clustering dendrograms for abundance profiles from real gut datasets.

Key observations:
- Tools of the same type clustered together in PCA space, indicating that tool category is a major driver of reported abundance profiles
- Kmer-based tools formed one cluster; mapping-based and general-purpose mappers formed a separate but nearby cluster; protein-DB tools formed a distinct outlier cluster
- The dendrograms confirmed the PCA patterns: within-category tools were more similar to each other than between-category tools
- Different gut samples showed consistent clustering patterns, suggesting the inter-tool variation is systematic and reproducible
- The discordance between tool categories on real data highlights the importance of using multiple approaches for sensitive experiments

## 16. Effect of Read Length on Accuracy

- Longer reads consistently improved species-level classification accuracy across all tools
- However, using only the longest reads (e.g., top quartile) reduced accuracy because species with naturally shorter reads become underrepresented
- The difference in read length distributions among species means that any length-based filtering introduces compositional bias
- PacBio HiFi reads, which have both high accuracy and relatively uniform lengths, showed the best classification performance overall
- ONT reads had more variable lengths and higher error rates, leading to more false positives

## 17. Protein vs Nucleotide Database Performance

- Protein-DB tools (Kaiju, MEGAN-P) consistently underperformed nucleotide-DB tools across all metrics: read-level accuracy, abundance estimation, and organism detection
- The protein database contained more tax IDs (11,435 vs 9,044) but this did not translate to better performance
- The lower specificity of protein-level matching (many conserved domains shared across species) likely contributed to higher false-positive rates
- The authors note that protein-DB tools might perform better on RNAseq data where transcript-level matching is more appropriate

## 20. References and Methodology Notes

- 45 references cited in the paper
- All tools were tested against custom-built databases from the same NCBI RefSeq sequences to eliminate database-content bias
- Abundance calculated using three definitions: relative read count, relative genome length, relative genome count
- Genome lengths scraped from NCBI website using median total length (Mb)
- Read classifications evaluated at both species and genus taxonomic levels
- Taxonomy extracted from nodes.dmp files from NCBI
- Reads assigned to higher taxa than species were counted as unclassified for species-level analysis
- MEGAN-N pipeline could not be run on PB3 and PB_Zymo datasets
- All datasets for resource benchmarking were normalized to approximately 1 Gbp (file size ~2 GB)
